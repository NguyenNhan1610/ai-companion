# Task Tracking (DAG-first)

Structured todo-list management with explicit typed dependencies, priority, confidence, effort estimates, and full traceability to upstream planning documents.

## Layout contract

A todo-list is a YAML file at `.project/todo-lists/TODO-{NN}-{slug}.yaml`. The file order is fixed so agents can read the graph before task bodies:

1. **Header** — `id`, `type`, `slug`, `title`, `status`, `upstream`, `downstream`, timestamps.
2. **DAG block** — `dependencies:` then `computed:`. These appear *before* `tasks:` so a reader loading the first ~60 lines gets the full graph.
3. **Task bodies** — `tasks:` list, one entry per task with metadata.

See `references/todo-schema.yaml` for the canonical example and the full validation rules.

## Commands

- `/ai:todo` — manage the todo-list (create, update, board view).
- `/ai:task-graph` — render the DAG (ASCII tree by default; `--format mermaid|json`; `--critical-path`; `--ready`).

## Task fields

| Field | Type | Notes |
|---|---|---|
| `id` | `task-NN` (kebab) | unique within file |
| `title` | string | human-readable |
| `status` | enum | `pending` \| `ready` \| `in-progress` \| `blocked` \| `in-review` \| `complete` \| `cancelled` |
| `priority` | enum | `P0` \| `P1` \| `P2` \| `P3` |
| `confidence` | float | 0..1; generator estimate, human-editable |
| `effort` | enum | `XS` \| `S` \| `M` \| `L` \| `XL` — points 1/2/3/5/8 |
| `acceptance_trace` | list of IDs | `[EAC-01, FAC-02, ...]` — short-form acceptance criteria ids |
| `evidence` | list | `{file, line?, kind}` where kind ∈ `implementation` \| `test` \| `doc` |
| `ticket` | string \| null | external tracker reference |
| `assigned_to` | string \| null | assignee |
| `notes` | string | free text |

## Dependency edges

```yaml
dependencies:
  - from: task-01
    to: task-03
    kind: hard          # hard | soft | data
    reason: task-03 consumes the page component created by task-01
```

- `hard` — `task-03` cannot enter `in-progress` until `task-01` is `complete`.
- `soft` — advisory ordering; tooling warns if `task-03` starts before `task-01`.
- `data` — like `hard`, plus `task-01` must produce evidence that `task-03` consumes.

## Computed fields

`computed:` is regenerated by `/ai:task-graph` or on Write by the todo agent. Fields:

- `roots` — tasks with no incoming edges.
- `ready` — tasks that are `pending`/`ready` and have no incomplete `hard`/`data` predecessors.
- `critical_path` — longest effort-weighted chain from any root to any leaf.
- `critical_path_points` — total effort points on the critical path.
- `parallel_tracks` — independent subgraphs starting from each root.

Do not hand-edit `computed:` — it will be overwritten.

## Flow

```
IMPL task DAG + EAC  ─►  /ai:todo --from IMPL-03  ─►  todo-list YAML
                                                           │
                                                           ├─► /ai:todo board   (kanban view)
                                                           ├─► /ai:task-graph   (DAG view)
                                                           └─► /ai:todo update  (status + evidence)
```

Status progression: `pending` → `ready` (auto when deps clear) → `in-progress` → `in-review` → `complete`. Or `pending`/`in-progress` → `blocked` / `cancelled`.

## Validation

Writing or updating a todo-list runs `planning-docs.mjs validate`:

- header fields present and well-typed
- `upstream` paths exist
- task ids unique; priority/effort/status/confidence well-formed
- every dependency references a declared task
- no cycles across any edge kind
